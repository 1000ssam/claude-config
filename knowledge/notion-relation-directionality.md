# 노션 관계(relation) 방향성 함정 — 양면처럼 보여도 별개 단방향일 수 있다

## 증상
두 DB(A↔B)를 잇는 관계가 "한 관계의 양면"처럼 보이지만, 실제로는 **A에 단방향 relation 1개 + B에 단방향 relation 1개**, 즉 서로 독립된 별개의 관계인 경우가 있다.
- 이때 각 쪽의 링크 수가 **크게 다를 수 있다**.
- 실제 사례(CRM 마이그레이션): 이벤트 DB `수강생`(→고객) = 80링크인데, 고객 DB `참여한 콘텐츠`(→이벤트) = **663링크(고객의 93%)**. 한쪽(`수강생`)은 세 이벤트 모두 정확히 25에서 잘린 부분 입력이었고, 권위 데이터는 반대쪽에 있었다.

## 원인
- 노션 relation은 생성 시 **dual_property(양방향 동기화)** 또는 **single_property(단방향)** 으로 만들 수 있다.
- single_property면 반대편 DB에 동기화 백릴레이션이 생기지 않는다. 그래서 양쪽 DB에 이름만 비슷한(예: `수강생` / `참여한 콘텐츠`) **서로 무관한 두 relation**이 따로 존재할 수 있다.
- API 스키마에서 `relation.type === 'dual_property'`면 `dual_property.synced_property_name`이 있고, `single_property`면 없다. 이걸로 양면 여부를 구분한다.

## 해결 / 체크리스트
1. 관계 데이터를 옮기거나 집계하기 전에 **양쪽 DB 모두에서 relation 속성을 확인**하고, 각 쪽의 **총 링크 수를 세어 비교**한다. 수가 다르면 별개 단방향 관계다.
2. 한쪽이 잘려 보이면(특히 25/50/100 같은 라운드 숫자에서 멈춤 → 표시 한도/부분 입력 정황) **링크 수가 많은 쪽을 권위 데이터로 채택**한다.
3. 손실이 걱정되면 **양쪽의 합집합(union)** 으로 복원한다(관계는 집합이라 중복은 자동 무시, 기존 링크 손실 없음).
4. 명세/지시가 특정 쪽을 지목해도, 그 쪽이 빈약하면 그대로 따르지 말고 **반대쪽 수치를 먼저 제시**하고 사용자 판단을 구한다.

## 교훈
명세가 가리킨 관계라고 무비판적으로 쓰지 말 것. 노션에서 "A의 관계"와 "B의 관계"는 이름이 대칭이어도 자동으로 같은 데이터가 아니다. **이관·집계 전 양면 링크 수 대조**가 1차 검증이다.
