# Tailwind v4: 임의값 변형(`min-[…]:`)과 명명 브레이크포인트(`md:`)가 같은 속성을 다투면 출현 순서/특이도로 뒤집힘 → 후행 `!`로 강제

## 문제
같은 CSS 속성을 타깃하는 **임의값 변형**(예: `min-[900px]:columns-2`)과
**명명 브레이크포인트**(예: `md:columns-1`)를 동시에 쓰면, 의도한 변형이 안 이기거나
거꾸로 적용된다. 데스크톱 좁은 단(컬럼)에서 `columns-4` 같은 유틸이 의도치 않게 살아남거나,
`md:`와 `min-[…]:`가 같은 속성을 다툴 때 결과가 예측과 어긋난다.

증상 표현: 반응형이 "가끔" 어긋난다 / 특정 폭에서만 깨진다 / 클래스를 추가해도 안 먹는다.

## 원인
Tailwind v4는 변형을 **CSS 레이어/소스 출현 순서**로 해석한다.
- 임의값 변형과 명명 브레이크포인트는 미디어쿼리 `min-width` 값이 서로 다를 수 있어,
  **두 규칙의 적용 범위가 겹치는(overlapping) 구간**이 생긴다.
- 그 겹치는 구간에서 둘의 **특이도(specificity)가 동일**하면, **나중에 선언된 규칙이 이긴다.**
- 내가 의도한 우선순위와 실제 소스 출현 순서가 불일치하면 그대로 깨진다.

즉 두 변형이 **비중첩(non-overlapping)** 조건이 아니면 "더 좁은/구체적인 게 이기겠지"라는
가정이 성립하지 않는다 — v4는 미디어쿼리 특이도를 자동으로 올려주지 않는다.

## 해결책
1. **브레이크포인트 조건을 비중첩으로 재정의** (권장, 명시적 > 영리한).
   같은 속성을 다투는 두 규칙의 적용 범위가 겹치지 않게 범위를 명시한다.
   예: 모바일 1열 / 데스크톱 2열을 겹치지 않는 구간으로 끊어 선언.
   겹치는 구간이 없으면 출현 순서에 의존할 일 자체가 사라진다.
2. **그래도 특정 유틸을 강제해야 하면 후행 `!`(Tailwind v4 important 문법)**로 특이도를 올린다.
   v4에서는 클래스 **뒤**에 `!`를 붙인다(v3의 `!mt-0` → v4는 `mt-0!`).
   - 예: `mt-0!`, `md:columns-1!`
   - important는 특이도를 확실히 올려 "이긴다"를 보장하지만 남발하면 추적이 어려워지므로
     비중첩 재정의가 안 될 때의 마지막 수단으로 쓴다.
3. **한 체계로 통일**: 같은 속성에 임의값 변형(`min-[…]:`)과 명명 변형(`md:`)을 섞지 않는다.
   둘 중 한쪽 체계로만 책임지게 하면 충돌 자체가 안 생긴다.

## 진단 절차
반응형이 어긋날 때 "감으로 클래스를 더하기" 전에:
1. **어느 변형이 어떤 `min-width`에서 같은 속성을 다투는지** 먼저 따진다
   (두 규칙의 미디어쿼리 적용 범위가 겹치는 구간이 있는가?).
2. 겹치면 → 비중첩으로 재정의.
3. 재정의가 어려우면 → 이겨야 하는 쪽에 후행 `!`.

## 적용 범위
Tailwind v4를 쓰는 모든 프로젝트. 특히:
- `min-[…]:` 임의값 변형 + `md:`/`lg:` 명명 브레이크포인트가 같은 속성
  (`columns-*`, `grid-cols-*`, `flex-*`, `margin`/`padding`, `display` 등)을 동시에 건드릴 때
- "데스크톱 좁은 단에서만 레이아웃이 깨진다" 류의 폭 의존 버그
- (확인: wee-linked.com 리디자인 중 다단 레이아웃에서 발생)

## 라우터 키워드
Tailwind v4 변형 순서, min-[] md 충돌, columns 의도치 않게, 후행 ! important, mt-0!,
반응형 어긋남 변형 특이도, 비중첩 브레이크포인트, 임의값 변형 명명 브레이크포인트 충돌
