# 뉴스레터 지표 추적 — 저장소 결정 메모 (Tier B)

> 2026-05-28 작성. 오픈·클릭·조회 추적의 **저장 백엔드 결정**이 구현의 선행 조건이라 먼저 정리.
> 결론만 보려면 맨 아래 "추천" 참조. 돌아오면 yes/no만 주시면 구현 들어감.

## 1. 추적 대상과 요구 특성

| 지표 | 트리거 | 빈도/패턴 | 핵심 요구 |
|------|--------|-----------|-----------|
| **오픈(open)** | 메일 내 1×1 트래킹 픽셀 GET | 발송 직후 수분 내 **스파이크**(수천 명 → 동시 수백 hit) | 빠른 원자적 증가, fire-and-forget |
| **클릭(click)** | 링크 래핑 → 리다이렉트 엔드포인트 | 산발적, 저지연 리다이렉트 필수 | 기록 후 즉시 302 |
| **조회(view)** | 웹 레터 페이지 방문 | 완만 | 단순 카운터 |

공통: **원자적 증가(INCR)** 가 필요. read-modify-write는 동시성에서 유실(lost update) 발생.

## 2. 후보 비교

| | Upstash Redis | Vercel KV | Notion DB |
|---|---|---|---|
| 원자적 INCR | ✅ 네이티브 | ✅ (= Upstash) | ❌ (RMW, 경쟁조건) |
| 지연시간 | ~수ms (REST) | ~수ms | **100~300ms+** |
| 레이트리밋 | 매우 높음 | 높음 | **~3 req/s (치명적)** |
| 서버리스 적합 | ✅ HTTP 기반, 커넥션풀 무관 | ✅ 네이티브 | △ |
| 무료 한도 | 10K cmд/day | Vercel 플랜 종속 | 사실상 부적합 |
| 사람이 보기 | 대시보드/CLI | Vercel 대시보드 | ✅ 노션에서 바로 |
| 신규 인프라 | 계정 1개 | 통합 추가 | 없음(기존) |

핵심 사실:
- **Vercel KV는 현재 Upstash 기반**(Vercel이 1st-party KV를 마켓플레이스 Upstash로 전환). 즉 "Vercel KV" ≈ "Upstash via 마켓플레이스". 기능 동일.
- **Notion은 핫패스에 부적합**: 3 req/s 한계 + 원자증가 없음 → 발송 직후 오픈 스파이크에서 즉시 레이트리밋·데이터 유실. 단, **집계 요약 저장**(캠페인 종료 후 opens=X/clicks=Y를 캠페인 페이지에 1회 기록)에는 적합.

## 3. 추천

**A안 (추천): Upstash Redis(핫패스) + Notion 롤업(선택)**
- 원시 이벤트는 Upstash에 원자 INCR로 빠르게 적재.
  - 키 설계 예: `open:{campaignId}` (총합), `open:{campaignId}:{subId}` (중복제거용 SET), `click:{campaignId}:{urlHash}`.
- 도입: **Vercel 마켓플레이스에서 Upstash 통합 추가** → `UPSTASH_REDIS_REST_URL`/`TOKEN` 자동 주입. 코드 한 줄로 연결.
- (선택) 캠페인 종료 시 합계를 해당 **Notion 캠페인 페이지 속성**(오픈수/클릭수/CTR)에 1회 기록 → 사람이 노션에서 바로 확인. 핫패스 아님 → 레이트리밋 무관.

근거: 카운터 워크로드엔 Redis가 정석. 서버리스/엣지에서 HTTP REST라 커넥션 이슈 없음. 노션 친화 프로젝트라 "롤업만 노션"이 UX·성능 균형.

**B안: Vercel KV 통합** — A안과 실질 동일(내부 Upstash). Vercel 콘솔에서 끝낸다는 점만 차이. A안과 택1.

**C안(비추): Notion 단독** — 구현 단순해 보이나 첫 대량발송에서 깨짐. 명시적으로 피할 것.

## 4. 구현 시 주의 (결정과 무관하게 기록)

1. **오픈율 신뢰도**: Apple Mail Privacy Protection(MPP)이 이미지 프리페치 → 오픈을 대량 인플레이트(2021~). **오픈율은 참고치로만**, 클릭률(CTR)을 1차 지표로. 보고서에 "오픈율은 MPP로 과대 측정될 수 있음" 주석 권장.
2. **클릭 추적 = 링크 래핑**: 본문 링크를 `/api/click?c={campaign}&u={urlHash}&s={sub}` → 기록 후 302. 리다이렉트 지연 최소화(엣지/Redis).
3. **프라이버시**: 픽셀·클릭 추적은 추적 고지 대상. 푸터에 한 줄 고지 검토.
4. **중복 제거**: "고유 오픈/고유 클릭"은 `{campaign}:{sub}` SET의 SADD 결과로 판정.
5. **봇 필터**: MPP·보안 스캐너의 프리페치 hit 제거 위해 User-Agent/타이밍 휴리스틱(발송 직후 <2초 hit 등) 고려.
