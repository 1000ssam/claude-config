**자동화를 만들고 싶은데, Make를 배워야 할까? 아니면 AI에게 코드를 짜달라고 할까?**

Make, n8n 같은 노코드 자동화 도구와 Claude Code 같은 바이브코딩 도구를 비용, 유연성, 유지보수 관점에서 비교합니다. 노션 자동화를 실제로 두 방식 모두 사용해본 경험을 바탕으로 정리했습니다.

# 두 세계의 자동화

노션에서 반복 업무를 자동화하려면 크게 두 가지 접근법이 있습니다.

**노코드 자동화**: Make, n8n, Zapier 같은 도구에서 블록을 드래그해서 워크플로를 만드는 방식.

**바이브코딩 자동화**: Claude Code, Cursor, Windsurf 같은 AI 코딩 도구에게 "이거 만들어줘"라고 말해서 코드를 생성하는 방식.

둘 다 "코딩을 직접 하지 않는다"는 공통점이 있지만, 결과물의 성격이 완전히 다릅니다.

# 노코드 자동화 (Make, n8n)

## 어떻게 작동하나

Make나 n8n에서는 트리거와 액션을 시각적으로 연결합니다. 예를 들어 "노션 DB에 새 항목 생성 → Slack에 알림 전송"이라면, 노션 트리거 블록과 Slack 액션 블록을 선으로 잇기만 하면 됩니다.

## 강점

**즉시 사용 가능합니다.** 연동할 서비스를 선택하고, 계정을 연결하고, 필드를 매핑하면 끝입니다. 코드 한 줄 없이 10분 안에 자동화가 작동합니다.

**시각적 디버깅이 가능합니다.** 워크플로의 각 단계에서 데이터가 어떻게 흘러가는지 눈으로 확인할 수 있습니다. 어디서 에러가 났는지 바로 보입니다.

**커뮤니티 템플릿이 풍부합니다.** "노션 + Slack" 자동화 같은 일반적인 케이스는 템플릿을 복사해서 바로 쓸 수 있습니다.

## 한계

**복잡한 로직에 약합니다.** 조건 분기가 3단계 이상 중첩되거나, 배열 데이터를 가공해야 하거나, 외부 API 응답을 파싱해서 재가공하는 작업은 노코드 블록으로 표현하기 어렵습니다.

**비용이 실행 횟수에 비례합니다.** Make 기준 무료 플랜은 월 1,000 operations, 유료 플랜은 $9/월부터 시작합니다. 자동화가 복잡해질수록 operation 수가 기하급수적으로 늘어납니다.

**커스텀 로직의 벽이 있습니다.** "학생 응답을 Claude API에 보내서 생기부 형식으로 변환한 뒤, 특정 형식으로 노션 DB에 기록"하는 것은 노코드로 가능은 하지만, HTTP 모듈로 직접 API를 호출하고 JSON을 파싱해야 합니다. 이 시점에서 노코드의 이점이 크게 줄어듭니다.

# 바이브코딩 자동화 (Claude Code 등)

## 어떻게 작동하나

AI 코딩 도구에게 자연어로 원하는 자동화를 설명하면, 코드를 생성해줍니다. 생성된 코드를 Vercel, Cloudflare Workers 등에 배포하면 자동화가 완성됩니다.

"노션 누가기록 DB에서 학생 데이터를 읽어서 Claude API로 생기부 초안을 생성하고, 결과를 같은 DB에 기록하는 서버리스 함수를 만들어줘."

이 한 문장으로 실제 작동하는 코드가 나옵니다.

## 강점

**유연성에 제한이 없습니다.** 코드이므로 어떤 로직이든 구현할 수 있습니다. 프롬프트 캐싱, 병렬 처리, 지수 백오프 재시도, 에러 시 Slack 알림 등 세밀한 제어가 가능합니다.

**실행 비용이 거의 없습니다.** Vercel 무료 플랜으로 충분하고, API 호출 비용만 발생합니다. Make처럼 실행 횟수에 따른 과금이 없습니다.

**코드를 직접 짤 필요가 없습니다.** AI가 코드를 생성하고, 타입 체크하고, 배포까지 도와줍니다. "바이브코딩"이라는 이름처럼 방향만 잡아주면 됩니다.

## 한계

**프로그래밍 개념을 이해해야 합니다.** 코드를 직접 작성하진 않지만, API, 환경변수, 서버리스 함수, 배포 같은 개념을 이해해야 AI와 대화가 됩니다. "코딩을 모르는 사람"이 바로 쓰기는 어렵습니다.

**시각적 디버깅이 없습니다.** 에러가 나면 로그를 읽어야 합니다. Make처럼 "여기서 실패했다"를 시각적으로 보여주지 않습니다.

**유지보수가 코드 수준입니다.** 자동화를 수정하려면 코드를 수정하고 재배포해야 합니다. 노코드처럼 블록을 드래그해서 바꿀 수 없습니다. 물론 AI에게 수정을 시키면 되지만요.

# 구체적인 비교

| 항목 | Make / n8n | 바이브코딩 |
|------|------------|------------|
| 진입 장벽 | 낮음 (드래그 앤 드롭) | 중간 (개념 이해 필요) |
| 구축 속도 | 단순 자동화: 빠름 | 복잡 자동화: 빠름 |
| 유연성 | 제공된 블록 범위 내 | 제한 없음 |
| 월 비용 | $9~$30+ (실행 횟수 비례) | $0~$5 (API 비용만) |
| 디버깅 | 시각적 | 로그 기반 |
| 유지보수 | 블록 수정 | 코드 수정 (AI 지원) |
| AI API 연동 | HTTP 모듈로 가능하나 번거로움 | 네이티브, 자유로움 |
| 확장성 | 플랜 제한 | 인프라 제한 (거의 없음) |

# 누가 뭘 써야 하나

<callout icon="💡" color="green_bg">
**노코드(Make, n8n)가 맞는 사람**
- 프로그래밍 개념에 익숙하지 않은 분
- "A가 발생하면 B를 실행"하는 단순한 자동화가 필요한 분
- 자동화 구축에 시간을 최소화하고 싶은 분
- 커뮤니티 템플릿으로 해결되는 일반적인 유스케이스
</callout>

<callout icon="💡" color="green_bg">
**바이브코딩이 맞는 사람**
- API, 서버, 배포 같은 개념을 어느 정도 이해하는 분
- 복잡한 데이터 가공이나 AI API 연동이 필요한 분
- 월 구독 비용을 최소화하고 싶은 분
- 자동화의 세부 동작을 직접 통제하고 싶은 분
</callout>

# 두 방식을 함께 쓸 수도 있다

반드시 양자택일할 필요는 없습니다.

단순한 알림 자동화("새 페이지 생성 → Slack 알림")는 Make로 5분 만에 만들고, 복잡한 처리("학생 데이터 → AI 가공 → DB 기록")는 바이브코딩으로 서버리스 함수를 만드는 식으로 병행할 수 있습니다.

중요한 건 도구에 맞춰 문제를 끼워 맞추지 않는 것입니다. **문제의 복잡도에 맞는 도구를 선택하면 됩니다.**

<callout icon="📝" color="blue_bg">
**시리즈 핵심 정리**

1편: 노션 워커스는 커스텀 에이전트에 코드를 심는 알파 기능이다
2편: 코드를 짤 수 있다면, 서버리스 함수가 크레딧과 안정성 면에서 유리하다
3편: 단순 자동화는 노코드, 복잡한 자동화는 바이브코딩이 적합하다
</callout>
