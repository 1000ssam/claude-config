# 자체 호스팅 뉴스레터 직접 구축 가이드

> 메일 발송 SaaS(메일리 등)를 직접 만든 시스템으로 대체하려는 사람을 위한 설계·의사결정·함정 정리.
> 코드를 통째로 주는 튜토리얼이 아니라, **"무엇을 왜 그렇게 했고, 어디서 막히는가"**를 정리한 지도입니다. 각 기능은 바이브코딩으로 직접 구현하는 걸 전제로 합니다.

<callout icon="🧭" color="blue_bg">
	이 문서의 한 줄 요약: 발송 자체는 전체 일의 1할입니다. 진짜 9할은 "도달률 · 발신 평판 · 동의/명단 관리"예요. 그래서 목차도 그 순서로 무게를 뒀습니다.
</callout>

---

## 0. 먼저 — 이게 당신에게 맞는가

### 비용 구조

| | SaaS(메일리 등) | 자체 구축(SES) |
|---|---|---|
| 과금 | 구독자 수 기준 월정액 | 발송 건수 실비 |
| 단가 | 구독자 늘수록 비례 증가 | **1,000통당 약 $0.10 (≈130원)** |
| 수백~수천 명 규모 | 매달 고정비 | 사실상 0에 수렴 |
| 대신 드는 것 | 없음(다 해줌) | **세팅·운영을 직접** |

### 언제 맞고, 언제 아닌가

- **맞다**: 발송 빈도가 낮고(주1~월1회), 규모가 수백~수천이고, 한 번 세팅하면 직접 운영할 의향이 있다.
- **아니다**: 매일 대량 발송, A/B 테스트·세그먼트·자동화 시퀀스가 핵심이다, 운영에 시간 못 쓴다 → SaaS가 싸게 먹힙니다.

<callout icon="⚠️" color="orange_bg">
	"발송 버튼만 만들면 끝"이라고 생각하고 시작하면 100% 후회합니다. 메일이 스팸함으로 가는 이유를 디버깅하는 데 대부분의 시간이 들어가요. 이 점을 받아들이고 시작하세요.
</callout>

## 1. 전체 구조

### 구성 요소

| 역할 | 도구 | 이유 |
|---|---|---|
| 발송 엔진 | **AWS SES** | 실비가 압도적으로 쌈. 대신 평판 관리는 내 몫 |
| 구독자 DB · 원고 저장소 | **Notion** | 이미 쓰던 도구. 새 관리 화면 만들 필요 없음 |
| 구독폼 · 해지 · 웹훅 | **Vercel (Next.js)** | 폼/콜백/웹훅 같은 서버 로직 한 곳에 |
| 발송 트리거 | **로컬 CLI 스크립트** | 발송은 가끔이라 버튼 UI보다 명령어 한 줄이 단순·안전 |
| 레이트리밋 · 지표 카운터 | **Upstash Redis** | 서버리스 환경에서 가벼운 카운터 |

### 데이터 흐름

```
[구독]  방문자 → 구독폼(Vercel) → 봇/동의 검증 → Notion 구독자 DB → 환영메일(SES)

[발송]  Notion 원고 → CLI 스크립트 → 1인1메일 개인화 → SES → 받은편지함
                                          ↓
                                   발송 로그(멱등) · 지표

[피드백] 반송/스팸신고 → SES → SNS → 웹훅(Vercel) → Notion 구독자 자동 정리
        오픈/클릭 → 추적 링크(Vercel) → Redis 카운터 → Notion 지표 동기화
```

<callout icon="💡" color="gray_bg">
	핵심 설계 원칙: "발송"과 "구독/피드백"을 분리했습니다. 발송은 내 PC에서 도는 CLI(트래픽 0, 실수해도 멱등 로그로 복구), 구독/웹훅은 항상 떠 있어야 하니 Vercel. 이 둘을 같은 서버에 묶지 않은 게 운영을 단순하게 만듭니다.
</callout>

## 2. 사전 준비 — 가장 많이 막히는 구간

> 코드보다 **외부 셋업(AWS·DNS)**에서 거의 다 막힙니다. 여기를 통과하면 절반은 끝.

### 2-1. AWS SES

1. AWS 계정 생성 → SES 콘솔(서울 리전 `ap-northeast-2` 권장)
2. **도메인 인증**(이메일 인증 말고 도메인 인증으로 — 발신 주소를 자유롭게 쓰려면 필수)
3. **프로덕션 액세스 신청**(샌드박스 탈출): 기본은 "인증된 주소로만 발송" 상태예요. 실제 구독자에게 보내려면 신청서를 내고 승인받아야 합니다. **승인에 하루~이틀 걸리니 가장 먼저 신청**하세요.

### 2-2. DNS 인증 4종 — 스팸함을 가르는 기본기

받는 쪽(지메일·네이버)이 "이 메일이 진짜 그 도메인이 보낸 게 맞나"를 검사합니다. 넷 다 세팅해야 해요.

| 레코드 | 역할 | 비고 |
|---|---|---|
| **SPF** | "이 서버가 우리 대신 보내도 됨" 허가 명단 | Custom MAIL FROM 도메인에 SES용 SPF |
| **DKIM** | 메일에 전자서명 → 위변조 아님 증명 | SES Easy DKIM = CNAME 3개 |
| **Custom MAIL FROM** | 봉투 발신 주소를 내 서브도메인으로 → SPF 정렬 | 예: `mail.내도메인.com` |
| **DMARC** | SPF/DKIM 불일치 시 처리 정책 | `_dmarc` TXT, 시작은 `p=none`(관찰) |

<callout icon="🇰🇷" color="red_bg">
	한국 레지스트라(가비아 등) 실전 함정 두 가지. 이거 모르면 며칠 날립니다.
</callout>

**함정 1 — SPF 값 앞 공백**: 가비아 TXT 입력창에 값 맨 앞 공백이 한 칸이라도 들어가면(`" v=spf1..."`) SES가 "SPF를 못 찾음"으로 인식합니다. 반드시 `v=spf1`로 시작하게, 앞 공백 0칸.

**함정 2 — 서브도메인 DKIM의 2-dot 호스트**: 발신 도메인을 `news.내도메인.com` 같은 서브도메인으로 격리하고 싶어도, 서브도메인 DKIM은 `<토큰>._domainkey.news` 형태의 점 2개짜리 호스트명이 필요한데 가비아가 이걸 못 받습니다. 외부 DNS(Route53 등) 위임 외엔 우회가 어려워요. **초기엔 루트 도메인으로 발송하고, 서브도메인 격리는 나중에 외부 DNS로 넘길 때 재검토**하는 게 현실적입니다.

### 2-3. Notion DB 3종

| DB | 핵심 속성 | 용도 |
|---|---|---|
| **구독자 DB** | 이메일, 이름, 상태(confirmed/unsubscribed/bounced/complained), 구독일시, 동의여부 | 명단 + 상태 관리 |
| **레터 DB** | 제목, 본문, 유형(일반/웰컴), 에디터, 발송일 | 원고 저장소 |
| **로그 DB** | 시각, 종류, 메시지 | 발송·동기화 실패를 눈에 보이게(fail-loud) |

## 3. 핵심 기능 — 구현 포인트와 함정

### 3-1. 구독 파이프라인

폼 제출 한 번에 여러 방어선을 **순서대로** 통과시키는 게 핵심입니다. 부수효과(DB 쓰기·메일 발송) 전에 차단해야 해요.

```
폼 제출
 → 1) IP 레이트리밋 (분당 N회)          ← DB/메일 건드리기 전
 → 2) 이메일 형식 검증
 → 3) 동의 여부 서버에서 강제 (fail-closed)
 → 4) 봇 검증 (Cloudflare Turnstile)
 → 5) 이메일 레이트리밋 (같은 주소 도배 방지)
 → 6) Notion 구독자 생성/갱신
 → 7) 환영메일 (best-effort: 실패해도 구독은 성공)
```

<callout icon="🔒" color="yellow_bg">
	동의 체크박스는 클라이언트를 절대 믿지 마세요. 프론트 체크박스는 우회 가능하니, 서버에서 consent===true를 다시 강제합니다(fail-closed). 동의 시각도 함께 기록해 두면 나중에 법적 근거가 됩니다.
</callout>

### 3-2. 옵트인 설계 — 단일 vs 더블

| | 단일 옵트인 | 더블 옵트인 |
|---|---|---|
| 흐름 | 제출 즉시 구독 확정 | 확인메일 클릭해야 확정 |
| 장점 | 전환율 높음, 단순 | 가짜/오타 주소 걸러짐, 명단 품질↑ |
| 단점 | 봇/오타 유입 가능 | 한 단계 이탈 발생 |

> 봇 차단을 캡차+레이트리밋으로 충분히 막는다면 단일 옵트인 + 환영메일 조합도 실용적입니다. 명단 품질이 평판에 직결되는 대량 발송이라면 더블 옵트인을 권장. **정답은 없고, 봇 방어 수준과 명단 민감도로 결정**하세요.

### 3-3. 발송 엔진 — 4가지가 필수

<callout icon="📨" color="blue_bg">
	1) 1인 1메일 개인화 — 받는 사람마다 개별 발송. 이름 인사 + 그 사람 전용 해지 링크. (한 통에 여러 명 BCC는 절대 금지: 평판·프라이버시 동시 사고)
</callout>

- **2) 멱등 발송 로그**: 매 건 발송 직후 로그에 기록. 중간에 끊겨 재실행해도 이미 보낸 사람은 건너뜀. 대량 발송 중 에러가 곧 재앙이 되지 않게 하는 안전장치.
- **3) 단계 발송(워밍업)**: 한 번에 전부 쏘지 않고 `50통 → 관찰 → 200통 → 나머지`. 신생 계정이 갑자기 대량 발송하면 스팸으로 찍힙니다.
- **4) 초당 레이트 제한**: SES 계정의 초당 송신 한도를 넘지 않게 보수적으로(예: 초당 10통). 동시성 + 레이트 게이트로 제어.

### 3-4. 도달률 3종 세트

| 항목 | 왜 필요 | 빠지면 |
|---|---|---|
| **List-Unsubscribe 원클릭 헤더** | 지메일이 요구하는 "한 번에 해지" | 스팸 점수 상승 |
| **HTML + 텍스트 멀티파트** | 텍스트 본문 없는 메일은 의심받음 | 스팸 점수 상승 |
| **이미지 영구 호스팅** | Notion presigned URL은 ~1시간 후 만료 | 메일 속 이미지 깨짐 → 스팸 신호 |

<callout icon="🖼️" color="gray_bg">
	이미지 함정: Notion에 올린 이미지의 URL은 임시(presigned, 약 1시간 만료)입니다. 그대로 메일에 박으면 한 시간 뒤 전부 깨져요. 발송 직전에 S3 등으로 재호스팅(영구 URL로 치환)한 뒤 보내야 합니다.
</callout>

### 3-5. 바운스 · 스팸신고 자동 정리

평판 보호의 핵심. 없는 주소로 계속 보내거나 신고한 사람에게 또 보내면 평판이 무너집니다.

```
SES 발송 → 반송/신고 발생 → SES가 SNS 토픽으로 통지
        → SNS가 내 웹훅(Vercel) 호출
        → 해당 구독자를 Notion에서 bounced / complained 로 자동 마킹
        → 다음 발송 대상에서 자동 제외
```

> SNS 구독은 처음 한 번 HTTPS 엔드포인트를 "Confirmed" 처리해야 작동합니다. 시뮬레이터 주소(`bounce@simulator.amazonses.com` 등)로 E2E 테스트를 꼭 해보세요.

### 3-6. 오픈 · 클릭 추적 (프라이버시 안전하게)

- 추적 URL은 전부 **HMAC 서명**을 달아 보냅니다. 서명 검증된 대상 URL로만 리다이렉트 → 오픈 리다이렉트(피싱 발판) 차단.
- 구독자 식별자는 **이메일의 해시(HMAC)**로. URL에 원문 이메일을 절대 노출하지 않습니다(고유 카운팅은 여전히 가능).

<callout icon="🪪" color="red_bg">
	링크에 이메일·이름 같은 개인정보(PII)를 절대 넣지 마세요. 해지 링크, 추적 링크, 웹뷰 링크 전부 비식별 토큰/해시로. 한 번 메일로 나간 링크는 회수 불가라 사고가 영구적입니다.
</callout>

### 3-7. 해지 처리

- 해지 토큰은 PII 없이, 서명된 비식별 ID로.
- 원클릭 해지(List-Unsubscribe-Post)와 웹 해지 페이지 둘 다 제공.
- 해지 즉시 Notion 상태를 `unsubscribed`로 → 다음 발송 제외.

## 4. 운영 — 보내고 나서가 진짜다

### 첫 대량 발송 절차

1. 깨끗한 명단으로 시작(죽은 주소·미동의 제거). SaaS에서 넘어오는 경우 검증된 기존 명단을 그대로 쓰는 게 유리.
2. **워밍업**: `50 → 관찰 → 200 → 나머지`. 각 단계 사이 반송/신고율 확인.
3. SES 평판 대시보드를 주시.

### 평판 임계값 (신생 계정 기준)

<callout icon="📊" color="orange_bg">
	반송률 5% 이상, 스팸신고율 0.1% 이상이면 위험 신호입니다. 저볼륨 초기엔 몇 건만 반송돼도 비율이 확 튀니 더 조심. 반송/신고는 계정 전체·롤링으로 집계되며, 환영메일 반송도 포함됩니다.
</callout>

### 명단 위생

- 반송/신고 자동 마킹(3-5)으로 불량 주소가 다음 발송 전에 빠지게.
- 재구독/해지 토글 abuse는 레이트리밋으로 억제.

## 5. 법적 체크리스트 (한국)

> 광고성 정보 발송은 정보통신망법 적용 대상입니다. 사고 나면 비용이 크니 발송 전 확인하세요.

- [ ] 광고성 메일 제목에 **`(광고)`** 표기
- [ ] 본문에 **전송자 정보**(상호·연락처)와 **수신거부 방법** 명시
- [ ] **개인정보처리방침** 페이지 제공(`/privacy`)
- [ ] 구독 시 **마케팅 수신 동의**를 명시적으로 받고 **시각 기록**
- [ ] 야간(21시~익일 8시) 광고 발송은 별도 동의 필요

## 6. 한계와 트러블슈팅

### 이 구조의 한계

- **답장 수신 없음**: 발송 도메인에 수신함(MX)이 없으면 구독자가 답장해도 반송됩니다. 답장을 받고 싶으면 reply-to나 수신 설정을 따로 마련해야 해요.
- **운영을 직접**: 평판 모니터링, 명단 위생, DNS 관리가 전부 내 일.
- **세그먼트·자동화 시퀀스**는 기본 제공 안 됨(직접 구현해야).

### 실전 함정 모음

| 증상 | 원인 | 해결 |
|---|---|---|
| SES가 "SPF 못 찾음" | TXT 값 앞 공백 | 앞 공백 제거, `v=spf1`로 시작 |
| 메일 속 이미지가 한 시간 뒤 깨짐 | Notion presigned URL 만료 | 발송 직전 S3 재호스팅 |
| 발신 주소 바꿨는데 그대로 | 배포 env 미반영 | 로컬 env뿐 아니라 배포 플랫폼 env도 변경 후 재배포 |
| 서브도메인 DKIM 등록 실패 | 가비아가 2-dot 호스트 거부 | 루트 도메인 사용 또는 외부 DNS 위임 |

### B. 구축 순서 체크리스트

- [ ] AWS 계정 + SES 프로덕션 액세스 **신청**(가장 먼저, 승인에 시간 소요)
- [ ] 도메인 인증 + DNS 4종(SPF/DKIM/MAIL FROM/DMARC)
- [ ] Notion DB 3종 생성
- [ ] 구독 파이프라인(폼+봇차단+레이트리밋+동의강제)
- [ ] 발송 엔진(개인화+멱등로그+워밍업+레이트)
- [ ] 도달률 3종(List-Unsubscribe+멀티파트+이미지 영속화)
- [ ] 바운스/신고 웹훅 + E2E 테스트
- [ ] 오픈/클릭 추적(HMAC, PII 비노출)
- [ ] 법적 체크리스트
- [ ] 워밍업으로 첫 발송

### C. 도메인/발신 주소 마이그레이션 메모

기존 SaaS에서 쓰던 발신 주소(예: `news@`)를 그대로 가져오면, **수신자 단위 인게이지먼트 이력**(주소록 등록, "이 발신자 잘 열어봄" 학습, 필터 규칙)을 이어받습니다. 알고리즘 평판은 도메인+IP 기준이라 로컬파트(`@` 앞)는 영향이 거의 없지만, 위 "사람이 알아보는 익숙함"은 실익이 있어요. 단, 발송 IP는 새로 시작이라 **워밍업은 여전히 필요**합니다.
