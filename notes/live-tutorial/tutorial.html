클로드코드로 만드는 러닝 챌린지 자동화

90분 시연 흐름 — 이미지 한 장으로 노션이 자동으로 채워지는 시스템을, 외운 명령어 없이 한국어 한 마디로 만들어 봅니다.

진행 상태

  • M0. 빈 폴더에서 시작
  • M1. 노션 API 연결 — 사이드 컷
  • M2. 자동화 시나리오 설계 — 플랜모드
  • M3. 스크립트 빌드 + 자동 문서화 (/init)
  • M4. CLI에서 라이브 동작 확인
  • M5. 깃허브 + Vercel — 한 줄로 배포
  • M6. 모바일에서 진짜 사용해 보기
  • 보너스. lazyweb으로 디자인 리뷰
  • 마무리. 회고 — 외운 명령어 0개

한 줄 미션

챌린저들이 러닝 앱 스크린샷 한 장만 올리면, 거리·시간·페이스가 자동으로 노션 데이터베이스에 채워지는 시스템을 만든다.

다섯 명의 챌린저(1000쌤·닝닝·윈터·지젤·카리나)가 매일 인증샷을 올린다. 손으로 데이터를 옮기는 건 귀찮다. 자동화가 답이다.

M0. 빈 폴더에서 시작

새 프로젝트 폴더를 만든다 — 단, 명령어를 외우지 않고 한국어로.

핵심 컨셉 — "아 쫌 해줘~~~"

클로드코드의 본질은 컴퓨터 명령을 사람 말로 통역해주는 것이다. mkdir, cd, touch 같은 단어를 외울 필요가 없다. 한국어로 의도만 전달하면 된다.

진행

한국어로 요청한다:

"프로젝트 폴더 running-challenge-3 만들어줘."

클로드는 내부적으로 다음을 실행한다:

mkdir running-challenge-3
cd running-challenge-3

명령어는 청중 화면에 보이지만, 시연자는 단 한 단어도 입력하지 않았다.

청중에게 한 컷

"이게 오늘 시연의 출발점입니다. 저는 한국어로 말했고, 클로드가 통역해서 컴퓨터한테 전달했어요. 이 패턴이 90분 동안 반복됩니다."

M1. 노션 API 연결 — 사이드 컷

메인 작업이 진행되는 동안, 새 세션을 띄워서 노션 API를 살짝 만져본다. curl 한 줄로 노션 데이터베이스의 스키마를 읽어오는 시연.

핵심 컨셉 1 — GUI vs API

노션이라는 같은 데이터 창고에 문이 두 개 있다.

사람은 notion.so(GUI 문)로 들어가고, 앱은 api.notion.com(API 문)으로 들어간다. 같은 창고, 다른 출입구.

행동 사람 (GUI) 앱 (API)
노션에 행 추가 클릭 → 사진 첨부 → 타이핑 HTTP POST + JSON 한 줄
결과 같은 행이 들어간다 같은 행이 들어간다

식당 비유: 손님용 입구 vs 배달앱 전산 연동. 둘 다 같은 주방으로 가는 주문.

핵심 컨셉 2 — 시크릿 격리

API 토큰 같은 시크릿은 절대 대화창에 입력하지 않는다.

.env.local 파일에 한 줄로 저장하고, .gitignore로 깃에서 제외한다. 명령어에는 $NOTION_TOKEN이라는 변수 이름만 등장한다. 실제 값은 사용 시점에만 메모리로 끌어올린다.

진행

새 세션에서:

"이 폴더 .env.local에 NOTION_TOKEN이 있어. 파일 직접 열지 말고, 변수로 처리해서 노션 DB 스키마 가져오는 curl 짜줘."

클로드가 만드는 명령:

source .env.local && curl \
  -H "Authorization: Bearer $NOTION_TOKEN" \
  -H "Notion-Version: 2026-03-11" \
  "https://api.notion.com/v1/data_sources/<DS_ID>"

응답 JSON에서 properties 필드가 노션이 자기 양식을 알려준 모습이다.

청중에게 한 컷

"토큰을 단 한 번도 화면에 띄우지 않았어요. 명령에는 변수 이름만 적혀 있고, 실제 값은 파일에 격리돼 있습니다. 이게 시크릿 다루는 표준입니다."

M2. 자동화 시나리오 설계 — 플랜모드

"이미지 → OCR → 노션 자동 입력" 시스템을 어떻게 만들지, 클로드와 같이 결정한다.

핵심 컨셉 1 — 결정 흐름 (1안 vs 2안)

1안 (노션 트리거) 2안 (웹 폼 직접) ✓
입력 인터페이스 노션 (행 만들기 + 첨부) 웹페이지 한 장
챌린저가 배워야 할 것 노션 사용법 URL 열기 + 사진
노션 권한 필요? 챌린저별 편집 권한 운영자만
단계 수 4단계 hop (취약) 3단계 단방향

2안 선택 이유: 챌린저는 노션을 안 켜도 된다. 노션은 결과 저장소로만 동작한다. 무거운 노션 앱을 안 거치니 모바일 UX가 가볍다.

핵심 컨셉 2 — 플랜모드

클로드는 코드부터 안 짠다. 먼저 계획을 세운다.

Shift+Tab 두 번으로 플랜모드 진입 → 클로드가 PRD 작성 → 시연자 검토 → 승인 → 빌드 시작.

진짜 일은 코딩이 아니라 결정이다.

진행

플랜모드에서 한 줄로 요청:

"러닝 앱 스크린샷에서 거리·시간·페이스를 추출해서 노션에 자동 입력하는 웹앱 만들어줘. 챌린저는 5명이고 본인이 누구인지 골라서 사진만 올리는 구조로."

클로드가 PRD를 펼친다:

  • 핵심 모듈 (lib/gemini.js, lib/notion.js, cli-upload.mjs)
  • API 엔드포인트 (api/upload.js, api/challengers.js)
  • 웹 UI (public/index.html)
  • 환경변수 4종

청중과 함께 검토 → "OK 이대로" → 빌드 시작.

청중에게 한 컷

"진짜 가치는 PRD를 받는 시점에 있어요. 코드는 클로드가 알아서 짭니다. 하지만 어떤 구조로, 어떤 결정으로 만들 건지 — 이건 사람이 검토해야 합니다."

M3. 스크립트 빌드 + 자동 문서화

PRD에 따라 클로드가 스크립트를 빌드한다. 빌드가 끝나면 /init으로 CLAUDE.md를 자동 생성한다.

빌드 결과물

파일 역할
lib/gemini.js 이미지 → JSON으로 바꾸는 두뇌
lib/notion.js JSON을 노션에 던지는 손
cli-upload.mjs 둘을 묶은 진입점

핵심 컨셉 1 — 스크립트가 LLM 한계 넘는 두 축

축 1. LLM 추론에 의존하지 않는 결정적 흐름

분기·반복·조건은 코드가 결정적으로 처리한다. "거리 5km 넘으면 A, 아니면 B" → 매번 LLM에 묻지 않는다. if (distance > 5) 한 줄.

축 2. 실제로 프로그램을 동작시킬 수 있다

파일 읽고 쓰기, API 호출, DB 입력 — 컴퓨터의 진짜 능력. 브라우저 챗봇 LLM은 텍스트만 만든다. 클로드코드는 코드를 만들고, 그 코드가 진짜 일을 한다.

핵심 컨셉 2 — /init과 CLAUDE.md

코드가 어느 정도 쌓인 폴더에서 /init을 실행하면, 클로드가 코드를 스캔해서 CLAUDE.md를 자동 생성한다.

README.md CLAUDE.md
누가 읽나 사람 (신규 협업자) 클로드 (매 세션마다)
톤 인사·홍보·가이드 작업 매뉴얼
주 내용 설치법, 라이선스, 스크린샷 폴더 구조, 명령어, 환경변수, 주의사항

다음 세션에 다른 사람이 이 폴더에 들어오면, 클로드는 이미 프로젝트 구조를 알고 있다.

진행

스크립트 빌드 완료 후 한 줄:

"/init"

클로드가 lib/, api/, public/ 등을 훑어 CLAUDE.md를 생성한다.

청중에게 한 컷

"README는 사람용 안내문, CLAUDE.md는 클로드용 매뉴얼. 둘 다 마크다운이지만 독자가 다릅니다. 이 한 번의 /init이 다음 세션, 다음 협업자, 다음 달의 나에게 보내는 편지가 됩니다."

M4. CLI에서 라이브 동작 확인

배포 전에, 스크립트가 실제로 작동하는지 터미널 한 줄로 검증한다.

진행

node cli-upload.mjs samples/nrc_phone.jpg 닝닝

5초 안에:

  1. Gemini가 이미지 분석
  2. 추출된 JSON이 출력
  3. 노션에 새 행 추가
  4. 노션 페이지 URL 응답

노션 데이터베이스 화면을 같이 띄워두면, 행이 라이브로 채워지는 모습이 보인다.

핵심 컨셉 — "엔진 먼저, 차체 나중"

배포는 엔진을 자동차 차체에 끼우는 일일 뿐이다.

엔진이 잘 돌아가는지부터 먼저 확인했고, 다음 단계는 차체를 만들어 사용자에게 넘기는 일이다.

만약 배포에서 환경변수·빌드 에러로 막혀도, 우리는 이미 내부 엔진은 작동한다는 증거를 가지고 있다. 시연이 실패하지 않는 안전망.

청중에게 한 컷

"지금 이 한 줄 명령으로 노션에 행이 채워졌어요. 외부에서 보면 마법 같지만, 내부적으로는 Gemini API + Notion API 호출 두 번. 그게 끝입니다."

M5. 깃허브 + Vercel — 한 줄로 배포

코드를 인터넷에 올려서, CLI 못 치는 사용자도 쓸 수 있게 만든다.

핵심 컨셉 1 — git ≠ GitHub

git GitHub
무엇 변경 이력 관리 도구 인터넷 코드 호스팅 서비스
어디서 내 노트북 인터넷
인터넷 필요? X O

둘은 별개의 도구. git은 로컬, GitHub은 그 git 저장소를 받아주는 인터넷 창고.

우리 폴더 안에 숨겨진 .git 디렉토리에 모든 변경 이력이 압축돼 있다. 이걸 GitHub에 그대로 복사하면 백업 끝.

핵심 컨셉 2 — "CLI 지원 도구는 모두 클로드가 부린다"

깃허브에는 gh CLI, Vercel에는 vercel CLI가 있다. 클로드는 자기 손가락 대신 이 CLI들을 두드린다.

사람이 클릭으로 하던 10단계가, 자연어 한 줄 → 클로드가 분해 → CLI 5~7번 호출로 압축된다.

CLI든 API든, 사용 통로가 있으면 클로드가 조종할 수 있다.

핵심 컨셉 3 — 출판사 비유
깃허브 원고 보관 창고
Vercel 받아서 인쇄·배포해주는 출판사
푸시 원고 새 버전 입고
배포 책이 서점 진열
URL ISBN — 누구나 그걸로 찾는다

진행

한 줄로 요청:

"이 코드 깃허브에 비공개로 올리고 Vercel 프로덕션에 배포해줘. 환경변수는 .env.local에서 가져다 써."

클로드가 자동 분해 (5~7단계):

  1. git init (이미 됐으면 skip)
  2. gh repo create <이름> --private --source=. --push
  3. vercel link --yes
  4. 환경변수 4개를 production에 등록
  5. vercel --prod

배포 끝나면 URL이 출력된다.

청중에게 한 컷

"저는 5개 명령어를 외운 적이 없어요. 클로드가 자기가 알아서 분해해서 실행했어요. CLI를 외우는 사람에서 자연어로 부리는 사람으로 점프하는 순간입니다."

M6. 모바일에서 진짜 사용해 보기

배포된 앱에 폰으로 접속해서, 러닝 앱 스크린샷을 진짜로 업로드해본다.

진행

  1. 폰 브라우저에서 배포 URL 열기
  2. 챌린저 선택 (예: 닝닝)
  3. 사진 첨부 → "노션에 기록" 탭
  4. 30초 안에 노션 데이터베이스에 새 행 등장

듀얼 화면 권장: 폰 미러링 + 노션 데이터베이스 동시.

청중에게 한 컷

"이 화면 두 개를 동시에 봐주세요. 왼쪽은 챌린저가 폰으로 사진 올리는 화면, 오른쪽은 노션에 자동으로 행이 채워지는 화면. 두 개가 같은 시간선에서 움직입니다. 지금까지의 모든 단계가 이 한 컷으로 결과를 냅니다."

보너스. lazyweb으로 디자인 리뷰

시간 여유가 있을 경우만.

"디자인이 너무 미니멀하다 → lazyweb MCP로 레퍼런스 자동 수집 → 개선 리포트 받기"까지의 흐름을 시연한다.

핵심 컨셉 — MCP

MCP (Model Context Protocol): 클로드가 외부 도구와 대화하는 표준 통로.

노션은 API로, GitHub은 CLI로 부렸다. 그런데 클로드 자체에 새로운 능력을 추가하려면? — MCP.

외부 서비스가 MCP 서버를 띄우면, 클로드는 그 서비스의 도구들을 자기 도구처럼 부른다.

lazyweb이 그런 서비스 — UI 스크린샷 데이터베이스를 뒤져서 레퍼런스를 찾아준다.

진행

"현재 디자인이 좀 별로야. lazyweb으로 비교 레퍼런스 찾아서 개선안 만들어줘."

클로드:

  1. 현재 디자인 캡처
  2. lazyweb MCP로 유사 화면 검색
  3. 6장 레퍼런스 다운로드
  4. 5가지 개선 아이디어 + Before/After 리포트 작성

(시간 여유에 따라 적용은 시연 후 작업으로 미뤄도 OK.)

청중에게 한 컷

"디자인 리뷰는 보통 디자이너 한 명이 며칠 걸려서 하는 일이에요. MCP를 통하면 5분 안에 레퍼런스 비교 + 개선안 도출까지 됩니다."
