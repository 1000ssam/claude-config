## 2026-03-24

### 프론트엔드 vs 백엔드

- **프론트엔드**: 브라우저에서 실행되는 코드. 화면 UI 담당.
  - **React**: UI를 만드는 JavaScript 라이브러리. 버튼, 드래그앤드롭, 미리보기 같은 화면 요소 생성
  - **Next.js**: React 위에 올라가는 프레임워크. 라우팅, 빌드, 배포를 쉽게 해줌
- **백엔드**: 서버에서 실행되는 코드. 무거운 연산, DB 접근, 파일 처리 담당.
  - **Python (FastAPI)**: 서버에서 PDF 열기, 문항 감지, 이미지 크롭 같은 실제 연산 처리

### 배포 플랫폼 비교 (Vercel vs Render vs Railway)

| | Vercel | Render | Railway |
|---|---|---|---|
| 최적 용도 | 프론트엔드 + 가벼운 API | 풀스택, Docker | 풀스택, DB 포함 |
| 무료 플랜 | 있음 | 있음 (콜드 스타트) | $5 크레딧 후 유료 |
| Python 백엔드 | 가벼운 것만 (10초 제한) | 자유로움 | 자유로움 |
| Docker | 미지원 | 지원 | 지원 |

**핵심**: Python에 무거운 라이브러리(PyMuPDF, numpy 등)가 들어가면 Vercel은 피하고 Render/Railway 사용.

### 콜드 스타트

- 무료 플랜 서버는 일정 시간(~15분) 요청이 없으면 잠듦
- 다음 요청 시 서버를 다시 깨우는 데 30초~1분 소요
- 깨어난 후에는 정상 속도
- 유료 플랜은 항상 켜져 있어 콜드 스타트 없음

### pdf.js vs PyMuPDF

- **pdf.js**: 브라우저에서 돌아가는 JavaScript PDF 라이브러리. 텍스트 추출이 부정확하고, 이미지 블록 감지 불가. 좌표계 변환도 복잡.
- **PyMuPDF**: Python PDF 라이브러리. `search_for()`, `get_text(clip=)`, `get_pixmap(clip=)` 등 정밀한 API 제공. 서버에서만 사용 가능.
- **교훈**: 같은 기능이라도 라이브러리 능력 차이가 크다. 브라우저에서 다 하려고 하지 말고, 무거운 건 서버로 보내는 게 맞다.

### Docker (도커)

- **한마디**: 앱을 실행 환경째 통째로 포장하는 상자
- **해결하는 문제**: "내 PC에서는 되는데 서버에서 안 돼" → Docker 컨테이너 안에 Python, 라이브러리, 코드를 모두 넣으면 어디서든 동일하게 실행
- **Dockerfile**: 상자를 만드는 레시피. "Python 3.12 설치 → 의존성 설치 → 코드 복사 → 실행" 같은 단계를 정의
- **바이브 코딩에서의 교훈**: 처음부터 Docker로 개발하면 배포 시 환경 문제가 0. 나중에 감싸려면 의존성 꼬여서 고생함. exam-cropper가 정확히 이 케이스 (Vercel→Render 이사)
- **지원 플랫폼**: Render, Railway, Google Cloud Run 등. Vercel은 Docker 미지원.

## 2026-03-25

### 데이터베이스 스택 계층 이해

| 계층 | 예시 | 역할 |
|------|------|------|
| **SQL** (언어) | SQL | DB에 명령하는 언어 (Structured Query Language) |
| **DB 소프트웨어** | PostgreSQL, MySQL, SQLite | SQL을 실행하는 프로그램 |
| **호스팅 서비스** | Supabase, AWS RDS, PlanetScale | DB를 클라우드에서 운영해주는 서비스 |
| **ORM** | Prisma, Drizzle | 코드에서 SQL을 대신 써주는 라이브러리 |
| **RLS** | PostgreSQL 기능 | 행(row) 단위로 접근을 제어하는 보안 기능 |

### Supabase 구조

- **핵심**: PostgreSQL DB + 부가 서비스 번들 (인증, 스토리지, 실시간 구독, Edge Functions)
- **PostgREST**: DB 테이블을 자동으로 REST API로 노출 → anon key만 알면 접근 가능 → RLS로 방어 필요
- **notion-flow 사례**: PostgREST를 안 쓰고 Prisma로 직접 연결하지만, PostgREST API가 기본으로 열려 있어서 RLS 적용 필요했음

## 2026-05-13

### 멱등성 (Idempotency)

- **한마디**: 같은 동작을 1번 하든 100번 하든 결과가 같음.
- **수학**: f(f(x)) = f(x). 한 번 적용한 결과를 또 적용해도 안 변함.
- **일상 비유**: 엘리베이터 버튼. 1번 눌러도 10번 눌러도 호출은 한 번만.
- **HTTP 메서드**:
  - `GET`, `PUT`, `DELETE` → 멱등 (같은 요청 반복해도 서버 상태 동일)
  - `POST` → 보통 비멱등 (반복하면 새 리소스가 또 생성)
- **왜 중요?**: 분산 시스템에선 같은 요청이 두 번 도착하는 게 흔함.
  - Vercel cron이 드물게 두 번 트리거
  - 노션 자동화 웹훅이 타임아웃 후 재시도
  - 운영 중 코드를 다시 트리거 (디버깅용 수동 호출)
  - 이때 부작용(SMS 발송, 결제 등)을 두 번 일으키면 안 됨 → 멱등성 보장 장치 필요.

### running-challenge-3 코드에서 멱등성 적용 예시

`api/cron/daily-morning.js`:
```js
// 같은 날 중복 발송 방지
const todayIsoStart = `${today}T00:00:00.000+09:00`;
const sentToday = await hasSuccessfulMessage(c.challengerId, '독려', {
  afterIso: todayIsoStart,
});
if (sentToday) {
  summary.encourageSkipped += 1;
  continue;
}
```

발송내역 DB를 조회해서 "이 챌린저에게 오늘 이미 '독려' 성공 발송한 적 있어?"를 확인. 있으면 스킵.

웹훅 쪽엔 더 강한 멱등성 장치가 있음 (`run.smsSent` 플래그):
```js
if (run.smsSent) {
  return res.status(200).json({ ok: true, skipped: 'already sent' });
}
// ... 발송 후
await markRunSent(pageId);  // 노션 페이지에 '문자발송완료=true' 마킹
```

### 멱등성의 범위 (헷갈렸던 부분)

- 멱등성 체크가 막는 범위는 **체크 조건**에 달려있음.
- 위 cron 코드의 `afterIso=오늘 00:00`은 "오늘 안에서만" 중복 방지.
- 5/13에 발송된 기록은 5/14 cron에선 검색 범위 밖이라 멱등성 차단 안 됨 → 다시 발송될 수 있음.
- 즉 "같은 날 두 번 트리거 방지"용이지 "한 사람당 평생 1통"용이 아님.

### 교훈

- 부작용(SMS 발송, 결제, 외부 API 호출 등)을 일으키는 코드에는 **반드시 멱등성 장치**를 만든다.
- 어떤 키(challengerId+오늘 / runId / messageId 등)로 중복을 판정할지 명확히 한다.
- 멱등성 키의 **시간 범위**도 의식한다 — "오늘만"인지 "영원히"인지.

## 2026-05-17

### 서버리스 과금 모델 — 종량제와 "코드 무게 기반" 과금

- **종량제(usage-based)**: 서버리스의 사실상 표준 과금 방식. 쓴 만큼만 낸다.
- **핵심**: "호출 횟수"가 아니라 **"코드가 실제로 한 일의 무게(= 컴퓨트 소비량)"**로 매기는 것이 본류다. 호출당 요금은 보통 부수적·소액.

#### 주요 플랫폼 비교

| 플랫폼 | 과금 단위 | 시간 측정 방식 |
|--------|-----------|----------------|
| **AWS Lambda** | 호출당 요금($0.20/100만 요청) + GB-초(할당 메모리 × 실행시간) | 벽시계 시간 (I/O 대기 포함) |
| **Cloudflare Workers** | 요청 수 + CPU-ms($0.02/100만 CPU-ms) | CPU가 실제로 도는 시간만 (I/O 대기 미과금) |
| **Vercel Functions** (Fluid compute, Active CPU) | Active CPU 시간($0.128/시간) + 프로비저닝 메모리(GB-시) | Active CPU만 (I/O 대기 미과금) |

#### 트렌드

- **"벽시계 시간 전체 과금" → "CPU가 실제로 도는 시간만 과금"**으로 정밀화되는 흐름.
- AI 호출처럼 응답 대기(I/O 대기)가 긴 작업의 비용을 크게 줄이려는 방향.

#### 서버리스 컴퓨트 과금 vs API 상품 과금 (헷갈리는 부분)

- 순수 **"호출 1회 = 고정요금(무게 무관)"** 모델은 서버리스 컴퓨트가 아니라 **API 상품**(외부 SaaS API, 지도/번역 API 등)의 과금 방식이다.
- 서버리스 컴퓨트는 "코드가 한 일의 무게"로, API 상품은 "호출 건수"로 매기는 게 일반적.

#### Notion Workers 사례

- **"1 run" 단위**로 과금하되, run 단가가 작업량(실행 시간·처리량)에 따라 변동.
- AWS의 GB-초를 'run·크레딧'으로 추상화한 셈.
- 특이점:
  1. 명시적 계산식 없이 추상화돼 있어 **비용 예측이 어렵다**.
  2. **결정론적 코드 과금과 LLM 추론 과금을 하나의 '크레딧' 풀로 통합**했다.

#### 알아둘 용어

- **GB-초**: 할당 메모리(GB) × 실행 시간(초). AWS Lambda의 컴퓨트 단위.
- **CPU-ms**: CPU가 실제로 일한 시간(밀리초). Cloudflare Workers 단위.
- **Active CPU**: Vercel이 쓰는, CPU가 실제로 도는 활성 시간 개념.
- **벽시계 시간(wall-clock) vs CPU 시간**: 벽시계 시간은 I/O 대기까지 포함한 전체 경과 시간, CPU 시간은 CPU가 실제 연산한 시간만. AI 응답 대기처럼 노는 시간이 길면 둘의 차이가 크게 벌어진다.
