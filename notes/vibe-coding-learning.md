## 2026-03-24

### 프론트엔드 vs 백엔드

- **프론트엔드**: 브라우저에서 실행되는 코드. 화면 UI 담당.
  - **React**: UI를 만드는 JavaScript 라이브러리. 버튼, 드래그앤드롭, 미리보기 같은 화면 요소 생성
  - **Next.js**: React 위에 올라가는 프레임워크. 라우팅, 빌드, 배포를 쉽게 해줌
- **백엔드**: 서버에서 실행되는 코드. 무거운 연산, DB 접근, 파일 처리 담당.
  - **Python (FastAPI)**: 서버에서 PDF 열기, 문항 감지, 이미지 크롭 같은 실제 연산 처리

### 배포 플랫폼 비교 (Vercel vs Render vs Railway)

| | Vercel | Render | Railway |
|---|---|---|---|
| 최적 용도 | 프론트엔드 + 가벼운 API | 풀스택, Docker | 풀스택, DB 포함 |
| 무료 플랜 | 있음 | 있음 (콜드 스타트) | $5 크레딧 후 유료 |
| Python 백엔드 | 가벼운 것만 (10초 제한) | 자유로움 | 자유로움 |
| Docker | 미지원 | 지원 | 지원 |

**핵심**: Python에 무거운 라이브러리(PyMuPDF, numpy 등)가 들어가면 Vercel은 피하고 Render/Railway 사용.

### 콜드 스타트

- 무료 플랜 서버는 일정 시간(~15분) 요청이 없으면 잠듦
- 다음 요청 시 서버를 다시 깨우는 데 30초~1분 소요
- 깨어난 후에는 정상 속도
- 유료 플랜은 항상 켜져 있어 콜드 스타트 없음

### pdf.js vs PyMuPDF

- **pdf.js**: 브라우저에서 돌아가는 JavaScript PDF 라이브러리. 텍스트 추출이 부정확하고, 이미지 블록 감지 불가. 좌표계 변환도 복잡.
- **PyMuPDF**: Python PDF 라이브러리. `search_for()`, `get_text(clip=)`, `get_pixmap(clip=)` 등 정밀한 API 제공. 서버에서만 사용 가능.
- **교훈**: 같은 기능이라도 라이브러리 능력 차이가 크다. 브라우저에서 다 하려고 하지 말고, 무거운 건 서버로 보내는 게 맞다.

### Docker (도커)

- **한마디**: 앱을 실행 환경째 통째로 포장하는 상자
- **해결하는 문제**: "내 PC에서는 되는데 서버에서 안 돼" → Docker 컨테이너 안에 Python, 라이브러리, 코드를 모두 넣으면 어디서든 동일하게 실행
- **Dockerfile**: 상자를 만드는 레시피. "Python 3.12 설치 → 의존성 설치 → 코드 복사 → 실행" 같은 단계를 정의
- **바이브 코딩에서의 교훈**: 처음부터 Docker로 개발하면 배포 시 환경 문제가 0. 나중에 감싸려면 의존성 꼬여서 고생함. exam-cropper가 정확히 이 케이스 (Vercel→Render 이사)
- **지원 플랫폼**: Render, Railway, Google Cloud Run 등. Vercel은 Docker 미지원.

### 데이터베이스 스택 계층 이해

| 계층 | 예시 | 역할 |
|------|------|------|
| **SQL** (언어) | SQL | DB에 명령하는 언어 (Structured Query Language) |
| **DB 소프트웨어** | PostgreSQL, MySQL, SQLite | SQL을 실행하는 프로그램 |

### Supabase 구조

- **핵심**: PostgreSQL DB + 부가 서비스 번들 (인증, 스토리지, 실시간 구독, Edge Functions)
- **PostgREST**: DB 테이블을 자동으로 REST API로 노출 → anon key만 알면 접근 가능 → RLS로 방어 필요
- **notion-flow 사례**: PostgREST를 안 쓰고 Prisma로 직접 연결하지만, PostgREST API가 기본으로 열려 있어서 RLS 적용 필요했음

## 2026-05-13

### 멱등성 (Idempotency)

- **한마디**: 같은 동작을 1번 하든 100번 하든 결과가 같음.
- **수학**: f(f(x)) = f(x). 한 번 적용한 결과를 또 적용해도 안 변함.
- **일상 비유**: 엘리베이터 버튼. 1번 눌러도 10번 눌러도 호출은 한 번만.
- **HTTP 메서드**:
  - `GET`, `PUT`, `DELETE` → 멱등 (같은 요청 반복해도 서버 상태 동일)
  - `POST` → 보통 비멱등 (반복하면 새 리소스가 또 생성)
- **왜 중요?**: 분산 시스템에선 같은 요청이 두 번 도착하는 게 흔함.
  - Vercel cron이 드물게 두 번 트리거
  - 노션 자동화 웹훅이 타임아웃 후 재시도
  - 운영 중 코드를 다시 트리거 (디버깅용 수동 호출)
  - 이때 부작용(SMS 발송, 결제 등)을 두 번 일으키면 안 됨 → 멱등성 보장 장치 필요.

### running-challenge-3 코드에서 멱등성 적용 예시

`api/cron/daily-morning.js`:
```js
// 같은 날 중복 발송 방지
const todayIsoStart = `${today}T00:00:00.000+09:00`;
const sentToday = await hasSuccessfulMessage(c.challengerId, '독려', {
  afterIso: todayIsoStart,
});
if (sentToday) {
  summary.encourageSkipped += 1;
  continue;
}
```

발송내역 DB를 조회해서 "이 챌린저에게 오늘 이미 '독려' 성공 발송한 적 있어?"를 확인. 있으면 스킵.

웹훅 쪽엔 더 강한 멱등성 장치가 있음 (`run.smsSent` 플래그):
```js
if (run.smsSent) {
  return res.status(200).json({ ok: true, skipped: 'already sent' });
}
// ... 발송 후
await markRunSent(pageId);  // 노션 페이지에 '문자발송완료=true' 마킹
```

### 멱등성의 범위 (헷갈렸던 부분)

- 멱등성 체크가 막는 범위는 **체크 조건**에 달려있음.
- 위 cron 코드의 `afterIso=오늘 00:00`은 "오늘 안에서만" 중복 방지.
- 5/13에 발송된 기록은 5/14 cron에선 검색 범위 밖이라 멱등성 차단 안 됨 → 다시 발송될 수 있음.
- 즉 "같은 날 두 번 트리거 방지"용이지 "한 사람당 평생 1통"용이 아님.

### 교훈

- 부작용(SMS 발송, 결제, 외부 API 호출 등)을 일으키는 코드에는 **반드시 멱등성 장치**를 만든다.
- 어떤 키(challengerId+오늘 / runId / messageId 등)로 중복을 판정할지 명확히 한다.
- 멱등성 키의 **시간 범위**도 의식한다 — "오늘만"인지 "영원히"인지.

## 2026-05-17

### 서버리스 과금 모델 — 종량제와 "코드 무게 기반" 과금

- **종량제(usage-based)**: 서버리스의 사실상 표준 과금 방식. 쓴 만큼만 낸다.
- **핵심**: "호출 횟수"가 아니라 **"코드가 실제로 한 일의 무게(= 컴퓨트 소비량)"**로 매기는 것이 본류다. 호출당 요금은 보통 부수적·소액.

#### 주요 플랫폼 비교

| 플랫폼 | 과금 단위 | 시간 측정 방식 |
|--------|-----------|----------------|
| **AWS Lambda** | 호출당 요금($0.20/100만 요청) + GB-초(할당 메모리 × 실행시간) | 벽시계 시간 (I/O 대기 포함) |
| **Cloudflare Workers** | 요청 수 + CPU-ms($0.02/100만 CPU-ms) | CPU가 실제로 도는 시간만 (I/O 대기 미과금) |
| **Vercel Functions** (Fluid compute, Active CPU) | Active CPU 시간($0.128/시간) + 프로비저닝 메모리(GB-시) | Active CPU만 (I/O 대기 미과금) |

#### 트렌드

- **"벽시계 시간 전체 과금" → "CPU가 실제로 도는 시간만 과금"**으로 정밀화되는 흐름.
- AI 호출처럼 응답 대기(I/O 대기)가 긴 작업의 비용을 크게 줄이려는 방향.

#### 서버리스 컴퓨트 과금 vs API 상품 과금 (헷갈리는 부분)

- 순수 **"호출 1회 = 고정요금(무게 무관)"** 모델은 서버리스 컴퓨트가 아니라 **API 상품**(외부 SaaS API, 지도/번역 API 등)의 과금 방식이다.
- 서버리스 컴퓨트는 "코드가 한 일의 무게"로, API 상품은 "호출 건수"로 매기는 게 일반적.

#### Notion Workers 사례

- **"1 run" 단위**로 과금하되, run 단가가 작업량(실행 시간·처리량)에 따라 변동.
- AWS의 GB-초를 'run·크레딧'으로 추상화한 셈.
- 특이점:
  1. 명시적 계산식 없이 추상화돼 있어 **비용 예측이 어렵다**.
  2. **결정론적 코드 과금과 LLM 추론 과금을 하나의 '크레딧' 풀로 통합**했다.

#### 알아둘 용어

- **GB-초**: 할당 메모리(GB) × 실행 시간(초). AWS Lambda의 컴퓨트 단위.
- **CPU-ms**: CPU가 실제로 일한 시간(밀리초). Cloudflare Workers 단위.
- **Active CPU**: Vercel이 쓰는, CPU가 실제로 도는 활성 시간 개념.
- **벽시계 시간(wall-clock) vs CPU 시간**: 벽시계 시간은 I/O 대기까지 포함한 전체 경과 시간, CPU 시간은 CPU가 실제 연산한 시간만. AI 응답 대기처럼 노는 시간이 길면 둘의 차이가 크게 벌어진다.

## 2026-05-25

### 왜 Vercel은 안 되고 GitHub Actions는 됐나 (실행시간 한도)

- 관건 = **작업이 걸리는 시간 vs 실행 환경의 최대 실행시간 한도**.
- bullet 사이트 publish는 **전체 리빌드라 ~124초** 동안 응답을 안 주고 연결을 붙잡는다 → 호출한 쪽이 그 시간을 계속 기다려야 한다.
- **Vercel Hobby 함수 = 최대 60초** → 직접 publish 호출하고 기다리면 60초에 강제 종료(타임아웃). (Vercel Pro였으면 300초라 됐을 것)
- **GitHub Actions job = 최대 6시간** → 124초쯤은 여유롭게 대기 가능. 그래서 느린 실행은 Actions가 담당.
- **핵심 = 역할 분리**: "느린 작업을 끝까지 기다리는 쪽"과 "빠르게 트리거만 하고 빠지는 쪽"을 나눈다.
  - 트리거(<1초): Vercel 함수 — 60초 한도에 안 걸림
  - 실행(124초): GitHub Actions — 시간 한도 넉넉
- 즉 "Vercel이 안 되는 것"이 아니라 "Vercel이 124초짜리를 **끝까지 붙잡는** 게 안 되는 것". 역할을 쪼개니 둘 다 쓰임.
- 연결 개념: 위의 '벽시계 시간 vs CPU 시간' — 여기선 벽시계(I/O 대기 포함 전체 경과) 기준 한도가 발목을 잡은 것.

### HTTP 헤더 (웹훅 설정의 "키 / 값"이 바로 이것)

- HTTP 요청을 **봉투**로 비유하면:
  - **URL** = 보내는 주소
  - **헤더(header)** = 봉투 겉면에 붙이는 쪽지들. 각 쪽지가 `이름: 값` 쌍.
  - **바디(body)** = 봉투 안에 든 편지(실제 데이터)
- 노션 웹훅 설정의 **키 = 헤더 이름**, **값 = 헤더 내용**.
  - 예: 키 `X-Trigger-Key` / 값 `<시크릿>` → 요청에 `X-Trigger-Key: <시크릿>` 헤더가 붙어 나간다.
- 용도: 받는 서버(Vercel 함수)가 그 헤더를 읽어 **요청이 정당한지(암호 일치) 검사** → 일종의 출입증. 틀리면 401(거부).
- `X-` 접두 = "표준 아닌 커스텀 헤더"라는 관례 표시. 이름은 임의로 정해도 됨(보내는 쪽·받는 쪽 약속만 맞으면).
- 자주 보는 표준 헤더: `Authorization`(인증 토큰), `Content-Type`(바디 형식, 예: application/json), `User-Agent`(클라이언트 종류).

### GitHub Actions도 "서버리스"인가? (FaaS vs 작업 실행기)

- 넓은 의미론 맞지만 **종류가 다르다**.
- **서버리스 함수(FaaS)** — Vercel 함수, AWS Lambda: **요청-응답형**. HTTP 요청 받아 즉시 결과 반환. 시작 빠름(ms), 고동시성. API 엔드포인트용.
- **서버리스 작업 실행기(배치/CI)** — GitHub Actions: **이벤트로 트리거되는 일회성 job**. 호출자에게 결과를 즉시 안 돌려줌(접수 204만, 결과는 로그/상태로 확인). 시작 느림(수십초), 실행 길게(최대 6h). 빌드·배포·스크립트용.
- **공통점(서버리스다운 점)**: 서버 직접 관리 X, 필요할 때만 실행, 안 쓰면 0으로, 사용량 과금.
- **차이**: 트리거(HTTP 요청 vs 이벤트), 결과 반환 여부, 시작속도(ms vs 수십초), 실행시간(짧음 vs 6h), 동시성(대량 동시 vs 러너 쿼터).
- **우리 조합 이유**: 느리고 가끔 도는 배포 = Actions가 잘하는 배치 작업 / 즉시 응답하는 빠른 입구 = Vercel 함수. 역할대로 나눔.
- **주의**: Actions를 범용 무료 컴퓨트처럼 상시·고빈도로 돌리는 건 본래 용도(자기 repo의 CI/CD)가 아니고 쿼터·약관 이슈가 있음. 가끔 도는 개인 자동화 트리거는 정상 범위.

### 왜 같은 이메일이 모바일에서 글꼴이 더 깔끔한가 (시스템 폰트 스택 + 화면 밀도)

- 글꼴을 파일로 박는 게 아니라 **"있는 것 중 앞에서부터 골라 써라"** 목록(폰트 스택)을 준다. 이메일 본문: `-apple-system, BlinkMacSystemFont, 'Apple SD Gothic Neo', 'Malgun Gothic', sans-serif`.
- **OS마다 깔리는 폰트가 다름** → 같은 메일도 다르게 보임:
  - 아이폰/맥: `Apple SD Gothic Neo`(애플 한글 폰트) — 정제됨, 깔끔.
  - 윈도우: 앞의 것들이 없어 **`맑은 고딕`** 으로 떨어짐 — 더 투박.
- 거기에 두 가지가 겹친다:
  1. **화면 밀도(ppi)**: 폰 ~400ppi+, 데스크톱 모니터 ~100~150ppi. 픽셀이 빽빽할수록 글자 가장자리가 매끄러움(레티나). **체감상 제일 큼.**
  2. **안티앨리어싱 엔진**: macOS/iOS는 두툼·부드럽게, 윈도우 ClearType은 다르게 → 같은 폰트도 OS별로 다른 느낌.
- **결론**: 모바일이 더 깔끔한 건 정상. 데스크톱(윈도우/맑은고딕)이 "최저 공통 수준".
- **이메일에선 웹폰트 강제 비추**: Gmail·네이버가 웹폰트를 거의 무시/제거 → 어차피 시스템 폰트로 떨어짐. 그래서 시스템 폰트 스택이 이메일에선 정답. (앱 UI는 반대 — 웹폰트로 통일 가능)
- 연결 개념: 앱 UI = 환경을 내가 통제(웹폰트 OK) / 이메일 = 받는 쪽 환경 제각각이라 "최저 공통"에 맞춰 안전하게. 같은 "폰트" 문제라도 매체가 통제권을 가른다.

### 이메일 도달률은 왜 "한 번 통과 ≠ 영구 통과"인가 (발신 평판 + 워밍업)

- **도달률(deliverability)** = 보낸 메일이 스팸함이 아니라 **받은편지함**에 들어가는 비율. **고정값이 아니라 매 발송마다 재평가**된다.
- 받는 쪽(Gmail·네이버)은 발신자에게 **평판 점수(sender reputation)** 를 매긴다 — 도메인·발신IP에 붙는 신용점수. 살아있는 값이라 오르내린다.
- **평판을 움직이는 신호들**:
  - **반송률(bounce)**: 없는/오타 주소로 보내면 hard bounce → 하락. (SES: 5% 넘으면 검토, 10% 위험)
  - **신고율(complaint)**: 수신자가 "스팸 신고" → 가장 치명적. (SES: 0.1% 넘으면 경고권 / Gmail도 신고율 높으면 스팸行)
  - **engagement**: 특히 지메일은 열람·클릭·삭제 패턴을 학습. 안 열고 지우면 점점 스팸으로.
  - **인증(SPF/DKIM/DMARC)**: 깨지면 즉시 하락. (있어야 본전)
- **워밍업(warm-up)**: 이력 얕은 도메인이 **갑자기 대량 발송**하면 필터가 의심한다. 3통 → 552통 점프 자체가 위험 신호. → 신규 발송은 **며칠에 걸쳐 점진 증량**.
- **테스트 통과 ≠ 대량 발송 예측**: 소수의 "내 주소(잘 여는 주소)"로 보낸 테스트가 inbox에 꽂힌 건 **기본 셋업(인증)이 정상**이란 신호일 뿐. 552명 대량 발송은 볼륨·engagement·리스트 품질이 전부 달라서 **별개의 시험대**.
- **리스트 위생(list hygiene)**: 이관·수집 리스트의 죽은/오타 주소 = bounce 유발 = 평판 깎임. 보내기 전에 정리.
- **모니터링**: bounce/complaint 피드백(AWS는 SNS 토픽으로 수신)을 받아야 "깜깜이 발송"을 면한다. 안 받으면 평판이 깎이는 줄도 모름.
- 연결 개념: 위 'GitHub Actions 대량 발송'과 이어짐 — **보낼 수 있는 것(기술)** 과 **받은편지함에 도착하는 것(평판)** 은 다른 문제. 기술 파이프라인이 완성돼도 평판은 따로 관리해야 한다.

### 아키타입(archetype) vs 레이아웃(layout) — 같은 거 아님

- **레이아웃** = 일반 개념. "요소를 화면 어디에 얼마 크기로 두느냐"(좌표·크기). 아무 배치나 다 레이아웃.
- **아키타입** = **목적에 묶여 이름 붙은, 재사용되는 레이아웃 패턴**. 예: "SWOT 2×2"는 단순 배치가 아니라 *강점/약점/기회/위협을 보여준다는 용도*에 고정된 패턴.
- 비유: 레이아웃 = 옷의 재단(공간 배치) / 아키타입 = "정장·트렌치코트"처럼 이름·용도가 붙은 카테고리.
- ppt-lab 구조에서: `layouts.json`의 `archetypes` 블록이 곧 "레이아웃 라이브러리". 각 아키타입 안의 `slots`가 실제 좌표(=레이아웃). 즉 **아키타입 모음 = 레이아웃 축**.
- 연결 개념: "재구축한다"는 건 결국 *아키타입들의 좌표를 우리 그리드 시스템에서 다시 생성*하는 것. 유형(2컬럼·SWOT 등)은 산업표준이라 저작권 대상 아님 → 좌표값·명명·큐레이션만 우리 것으로 바꾸면 출처가 끊김.

### DB(데이터베이스) vs 테이블(table) — 노션 용어와의 충돌

- 헷갈림의 정체: **노션이 "데이터베이스"라 부르는 표가, SQL에서는 "테이블"** 이다. SQL의 "데이터베이스"는 그보다 한 단계 위(표들을 다 담는 통).
- 노션 ↔ SQL 매핑:
  | SQL 용어 | 노션에서는 | 한마디 |
  |---|---|---|
  | 데이터베이스(DB) 1개 | 워크스페이스(큰 페이지) 1개 | 모든 걸 담는 큰 통 |
  | 테이블 N개 | 노션 "데이터베이스(표)" N개 | 통 안의 개별 표들 |
- 즉 "DB 1개에 테이블 13개" = "노션 워크스페이스 1개 안에 노션 DB(표) 13개". 게시글 표·댓글 표·회원 표를 따로 두고 relation으로 잇는 것 = 노션에서 평소 하던 그대로.
- 왜 한 표에 안 몰고 쪼개나: 노션에서도 회원 명단과 댓글을 한 표에 안 섞음(성격·열이 다르니까). 같은 이유. 한 통에 다 욱여넣으면 엉망.
- wee-linked 실제 구성(테이블 13): posts(게시글)·comments(댓글)·boards/board_categories(게시판)·likes·bookmarks·reports(커뮤니티) + columns(칼럼) + resources/resource_categories(자료실) + profiles/verification_requests(회원·인증) + site_content(정적 문구).
- 운영 함의: 기능 전부가 **DB 1개**에 있으니, 그 DB가 정지되면 사이트 전 기능이 한 번에 멈춤. 반대로 keep-alive 핑 1개로 통 전체를 보호. (Supabase 프로젝트 1개 = DB 1개)
- 연결 개념: 위 '데이터베이스 스택 계층'(2026-03-25)에서 PostgreSQL은 DB 소프트웨어, Supabase는 호스팅. 그 PostgreSQL DB '한 통' 안의 칸막이가 테이블.

### OAuth vs SSO vs "공유 인증 백엔드" — 셋 다 다름

- **OAuth** = 권한 위임 **프로토콜(방법)**. "○○로 로그인" 버튼이 이것. 앱 B가 앱 A에게 "이 사람 너희 회원 맞아?" 물으면 A가 **서명한 쪽지**를 B에 보내주는 약속.
- **SSO(Single Sign-On)** = **목표/경험**("한 번 로그인 → 여러 앱 통함"). 방법이 아니라 결과. 보통 OAuth/OIDC로 구현. 즉 SSO ⊃ OAuth(구현 수단).
- **공유 인증 백엔드** = 두 앱이 **같은 회원 DB(같은 auth.users)** 를 직접 봄. 명부가 하나라 위임 쪽지(OAuth) 자체가 불필요. SSO 경험을 주되 federated SSO보다 단순.
- 비유(아파트): OAuth=건물 둘+각자 명부, A가 B에 "회원 맞음" 서명쪽지 전달 / 공유 백엔드=건물 하나·명부 하나, 웹문·앱문 어느 쪽으로 와도 같은 명부 조회.
- 헷갈림 포인트: "A로 OAuth 한다"와 "SSO"는 배타적이 아님. SSO를 OAuth로 만드는 게 흔한 조합. 단 **명부를 합치면 둘 다 안 거치고도 SSO 경험이 나옴**.
- 위링↔링글 실제 선택: 별도 프로젝트(=OAuth/federation)가 아니라 **위링 Supabase 프로젝트 공유**(§17.8). 이유 = Supabase는 프로젝트당 auth.users 1세트뿐이라 native 크로스-프로젝트 SSO 없음 → 합치는 게 FK 유지·끊김 없음. 로그인 수단: 웹=이메일/비번, 앱=이메일 OTP(일회용 코드) 우선, 구글 후순위.
- 연결 개념: 위 'DB vs 테이블'과 직결 — auth.users도 결국 그 한 DB 안의 한 테이블. DB를 공유한다 = 그 회원 테이블을 공유한다.
